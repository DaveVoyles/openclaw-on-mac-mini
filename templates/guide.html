1. Overview & Architecture

OpenClaw is a Discord bot that acts as an AI-powered operations agent for your Mac Mini Docker stack. It connects to:

  • Google Gemini 2.5 Flash — AI reasoning with function calling (66 skills) — used for tool-requiring, weather, and research queries
  • Ollama gemma3:12b — local LLM running on the Mac Mini for conversational queries (free, no rate limits, ~15–20 tok/s on M4 Neural Engine)
  • Docker Engine — manage 26+ containers running on the Mac Mini
  • *arr Stack — Sonarr, Radarr, Lidarr, Prowlarr, Bazarr via their APIs
  • Download Clients — SABnzbd (Usenet) and qBittorrent (torrents)
  • Plex / Tautulli — media server monitoring
  • Synology NAS — storage backend

How It Works — Hybrid Routing

You type /ask "hello"
    ↓
OpenClaw checks: does this need tool use? → No
    ↓
Routed to Ollama (local, free, <200ms)
    ↓
You get a fast conversational reply

You type /ask "What's downloading right now?"
    ↓
OpenClaw checks: does this need tool use? → Yes ("download" keyword)
    ↓
Sent to Gemini 2.5 Flash with 27 tool declarations
    ↓
Gemini calls get_download_queue
    ↓
OpenClaw queries SABnzbd + qBittorrent APIs
    ↓
Gemini formats the results into a human-readable answer
    ↓
You get a Discord embed with your active downloads

Gemini can chain up to 5 tool calls per question. Simple chat goes to Ollama at zero cost. The response footer shows which model answered.

3. The /ask Command (AI Chat)

The most powerful command. Uses hybrid routing: conversational queries go to local Ollama (free, fast), tool-requiring queries go to Gemini 2.5 Flash with function calling.

Example Queries

/ask What containers are running?
/ask Is Sonarr healthy? Any errors in the last 50 lines?
/ask Show me active downloads
/ask What was recently added to Plex?
/ask Run a speed test

💡 Context is per-channel: /ask automatically remembers the last 20 messages in a channel for 30 minutes. No need to reply-thread — just keep using /ask. Use /clear to reset, or /save to keep it permanently.

4. Conversation Threads (Save & Resume)

By default, /ask keeps 30 minutes of context in memory. If you're researching something and want to pick it up later, save the conversation as a named thread. Threads are written to disk and survive container restarts.

Workflow

1. Chat with /ask as normal
   /ask Tell me about the Sonarr upgrade options
   /ask What version is Sonarr on?
   /ask What would break if I upgraded?

2. Save when you want to stop:
   /save sonarr-upgrade
   → ✅ Saved thread sonarr-upgrade (6 messages)

3. Come back later (any time, even after restart):
   /resume sonarr-upgrade
   → ✅ Resumed thread sonarr-upgrade — 6 messages (saved 2026-03-23 14:30)

4. Continue the conversation:
   /ask OK where were we?

Commands

CommandWhat it does
/save <name>Snapshot current conversation to disk. Name: letters, digits, - or _, up to 32 chars.
/resume <name>Load a saved thread and make it the active conversation in this channel.
/threadsList all your saved threads with message count and save date.
/forget <name>Permanently delete a saved thread from disk.
💡 Thread names are per-user and stored in /memory/threads/. They are private to you and not visible to other bot users.
💡 Overwrites work: Running /save same-name again replaces the previous save — useful to update a thread you're actively working on.

5. Docker & System Commands

/containers

Lists all running Docker containers with their name, status, and port mappings. Quick way to see what's up.

/status <service>

Deep-dive on one container: CPU usage, memory, network I/O, ports, restart count, image version.

/status sonarr
/status qbittorrent
/status openclaw

/logs <service> [lines]

View the last N lines of a container's logs (default: 30, max: 100). Great for debugging.

/logs sonarr 50
/logs sabnzbd

/system

Mac Mini resource usage: CPU %, memory (used/total), disk space. Pulls from Glances if available, falls back to system commands.

/dockerstats

Per-container resource table: CPU %, memory usage, network RX/TX. Similar to docker stats but formatted nicely.

/restart <service> HIGH RISK

Restarts a container. This requires approval — you'll get a button prompt:

/restart sonarr
  → "Restart sonarr? ✅ Approve | ❌ Deny"
  → Click ✅ to confirm (expires in 5 minutes)

Protected services that can never be restarted via the bot: traefik, socket-proxy, homepage, watchtower.

8. Long-Term Memory (QMD)

OpenClaw has a persistent memory system that survives restarts. Store facts, preferences, IPs, credentials notes, or anything else you want the bot to remember.

/remember <content> [tags]

Store a fact. Tags are optional comma-separated labels for easier recall.

/remember The NAS IP is 192.168.1.8 tags:network,infrastructure
/remember Sonarr API key rotated on 2026-03-15 tags:sonarr,security
/remember Dave prefers dark mode dashboards tags:preferences
/remember qBittorrent login: admin / check-secrets-env tags:credentials

/recall <query>

Search your stored memories by keyword or tag.

/recall NAS
/recall sonarr
/recall credentials
💡 Tip: The AI can also use memory! When you /ask a question, Gemini can call recall_fact to check if it already knows the answer from stored memories. Try: /ask What do you remember about the NAS?

How It Works Under the Hood

Memories are stored as JSON in /memory/qmd.json. Each entry has a timestamp, content, and tags. Search is case-insensitive keyword matching against both content and tags.

9. Scheduled Tasks

Automate recurring operations. The scheduler runs any registered skill on a daily or interval basis.

/schedule list

View all scheduled tasks with their status, last run time, and run count.

/schedule add

Create a new scheduled task. Two modes:

Daily Schedule (specific time)

/schedule action:add skill:check_arr_health hour:6 minute:0
  → Runs check_arr_health every day at 6:00 AM

/schedule action:add skill:create_status_report hour:8 minute:30
  → Daily status report at 8:30 AM

Interval Schedule (every N minutes)

/schedule action:add skill:check_download_clients interval:30
  → Checks download clients every 30 minutes

/schedule action:add skill:get_docker_stats interval:60
  → Docker stats snapshot every hour

/schedule remove

/schedule action:remove task_id:sched-1

/schedule toggle

Enable or disable a task without deleting it.

/schedule action:toggle task_id:sched-1
💡 Tip: Good scheduled tasks to set up:
  • check_arr_health every 30 min — catch service issues early
  • create_status_report daily at 8 AM — morning briefing
  • get_docker_stats every 60 min — resource usage baseline

14. Deep Research (/research)

Multi-step autonomous research. Given a question or topic, the bot:

  1. Plans a set of focused sub-queries using Gemini
  2. Searches each sub-query in parallel (3-tier cascade: Tavily → DuckDuckGo → Bing)
  3. Browses the top source URLs to extract full article content
  4. Synthesizes a structured report using low-temperature Gemini in a new Discord thread

/research <query>

/research Best neighborhoods in Philadelphia under $400k with good schools
/research Docker vs Podman tradeoffs for home lab usage
/research How to set up Tailscale exit nodes on Synology NAS
/research Current Sonarr v4 migration breaking changes
💡 Live progress: A Discord thread is created immediately. The bot posts progress messages as each search and URL is processed. The final report appears with action buttons:
  📌 Save to Memory — stores the report summary in QMD long-term memory
  🔄 Re-run in 24h — schedules the research query to repeat tomorrow

15. Session Memory & Cross-Session Recall

OpenClaw automatically summarizes conversations and injects context into new sessions.

How It Works

  1. You have a conversation with /ask (4+ turns)
  2. After 30 minutes of inactivity the session expires
  3. Gemini auto-summarizes the session (~300 tokens)
  4. Summary saved to disk and QMD memory (/memory/summaries/{user_id}_last_session.json)
  5. Next conversation: the bot injects “Last time we talked about…” into context automatically
💡 No action needed: This is fully automatic. To query it explicitly:
/ask What did we talk about last time?

For facts you always want the bot to retain (not just session summaries), use /remember and /recall — see Section 8.

16. Autonomous Agent Operations

OpenClaw includes dedicated tools for long-running, multi-step tasks that require planning, research, and persistent working memory. These tools enable the AI to work autonomously on complex goals (Manus-style) without constant user prompting.

The Planning Workflow

When you give a complex goal via /ask (e.g., "Research and document a full upgrade path for our Docker stack"), the AI uses three persistent files to track its progress:

  1. task_plan.md: The master checklist of phases and steps.
  2. findings.md: Raw data, command outputs, and research notes.
  3. progress.md: A high-level executive summary of current status.

This "working memory on disk" allows the AI to maintain context over long periods and across multiple /ask sessions.

Core Autonomy Tools (via /ask)

  • init_planning_files: Creates the base plan and finding files for a new goal.
  • update_plan_status: Marks phases as complete and updates the current progress note.
  • autonomous_loop: Starts a continuous execution cycle where the AI checks its plan, performs the next action, evaluates the result, and loops until completion.

Controlling Autonomous Loops

Safety is built into the autonomous execution system:

  • Emergency Stop: If a file named STOP_AUTONOMOUS is created in the data/ directory, all loops immediately terminate.
  • Iteration Limit: Loops have a hard-coded maximum number of steps to prevent runaway resource usage.
  • Manual Resume: You can always check progress.md to see where the agent left off and resume the task later.

19. Power User Tips

🎯 Best Practices

  1. Use /ask for complex queries. Instead of running 5 separate commands, ask: "Check all services, show downloads, and tell me if anything looks wrong."
  2. Save research threads. Working on an upgrade or investigating an issue? /save my-topic before you stop. /resume my-topic to pick up where you left off.
  3. Store important info in memory. /remember API keys, server IPs, rotation dates. The AI can recall them later.
  4. Set up scheduled health checks. /schedule add skill:check_arr_health interval:30 catches issues before you notice them.
  5. Use /analyze for debugging. When a service acts up, /analyze sonarr 100 feeds 100 lines of logs to AI for analysis.
  6. Check /spending weekly. At ~$0.0005 per query, it'll last a long time, but it's good to monitor.
  7. Conversation threads for research. Save a thread, come back days later, resume it and the AI has full context from where you left off.
  8. Conversation context is your friend. Ask a question, then follow up: "What caused that error?" or "Show me the logs for that service."

21. API Gateway (Maton)

The API Gateway gives OpenClaw access to 100+ third-party APIs — Slack, GitHub, Google Workspace, Notion, HubSpot, Stripe, Airtable, and more — through a single Maton API key with fully managed OAuth. No need to handle OAuth flows yourself; Maton stores and refreshes the tokens.

How It Works

You: /ask Post “Deployment done” to Slack channel #deployments
    ↓
Gemini calls gateway_request(app="slack", path="api/chat.postMessage", ...)
    ↓
OpenClaw sends: POST https://gateway.maton.ai/slack/api/chat.postMessage
                Authorization: Bearer $MATON_API_KEY
    ↓
Maton injects the stored Slack OAuth token and proxies the request
    ↓
Slack receives the message — reply appears in Discord

The gateway base URL is https://gateway.maton.ai/{app}/{native-api-path}. You use native API paths exactly as documented by each service — Maton just handles auth.

Setup

  1. Sign in at maton.ai → Settings → copy your API key.
  2. It’s already in your .env as MATON_API_KEY. ✓
  3. Connect each app you want to use (only needs to be done once per app):
    /ask Create a Maton OAuth connection for slack
    → OpenClaw returns a URL → open it in your browser → complete OAuth → done.

The Three Gateway Skills

SkillWhat it does
gateway_requestCall any native API endpoint through the proxy. Supports GET, POST, PUT, PATCH, DELETE with JSON bodies. Returns the raw API response.
gateway_list_connectionsList all active OAuth connections (optionally filtered by app name).
gateway_create_connectionCreate a new OAuth connection for an app. Returns the OAuth authorization URL to open in a browser.

Rate Limits & Notes

  • Maton allows 10 requests/second per account.
  • Target API rate limits (e.g., Slack’s own limits) apply on top of this.
  • Custom headers and query parameters are forwarded to the target API.
  • All HTTP methods are supported: GET, POST, PUT, PATCH, DELETE.
  • If you have multiple OAuth connections for the same app, specify a connection_id in your request.
💡 Tip: You don’t need to know the native API URLs by heart — just describe what you want and Gemini will figure out the right endpoint. E.g. “/ask get the last 10 emails from Gmail via the gateway”.
